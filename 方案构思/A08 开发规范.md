## 二、Git 协作规范

### 2.1 分支命名规范（简化版）

```
main                # 主分支，保护分支，仅用于发布
dev                 # 开发分支，日常开发推送至此
```

> **注意**：MVP 阶段使用简化工作流，不需要 feature/fix 等分支，直接在 dev 开发推送。

### 2.2 提交信息规范

```
<类型>: <简短描述>

[可选：详细描述]

[可选：关联 Issue]
```

**类型说明**：

| 类型 | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat: 添加能耗查询接口` |
| `fix` | 修复 Bug | `fix: 修复 COP 计算错误` |
| `docs` | 文档更新 | `docs: 更新 API 文档` |
| `style` | 代码格式 | `style: 格式化代码` |
| `refactor` | 重构 | `refactor: 重构查询模块` |
| `test` | 测试 | `test: 添加单元测试` |
| `chore` | 构建/工具 | `chore: 更新依赖版本` |

**示例**：

```bash
# 好的提交信息
git commit -m "feat: 添加建筑能耗查询接口"
git commit -m "fix: 修复时间范围筛选 bug"

# 不好的提交信息
git commit -m "更新代码"
git commit -m "fix bug"
git commit -m "asdfasdf"
```

### 2.3 Git 工作流程（简化版）

```bash
# 每个人都直接在 dev 分支开发

# 1. 开始开发前
git checkout dev
git pull origin dev

# 2. 开发并提交
git add .
git commit -m "feat: 添加能耗查询接口"

# 3. 直接推送到远程 dev（无需 PR）
git push origin dev
```

> **简化原则**：MVP 阶段不追求复杂流程，能跑起来最重要。main 分支仅用于最终版本发布。

### 2.4 Git 配置（首次使用）

```bash
# 所有队员必须执行
git config --global user.name "你的 GitHub 用户名"
git config --global user.email "你的 GitHub 邮箱"
git config --global core.autocrlf input  # Linux/WSL
git config --global init.defaultBranch main
```

def calculate_cop(outlet_temp: float, return_temp: float, energy: float) -> float:
    """
    计算 COP 能效比
    
    Args:
        outlet_temp: 出水温度
        return_temp: 回水温度

**命名规范**：

| 类型 | 命名方式 | 示例 |
|------|---------|------|
| 变量/函数 | 小写 + 下划线 | `building_id`, `query_energy()` |
| 类名 | 大驼峰 | `EnergyQuery`, `DataProcessor` |
| 常量 | 全大写 + 下划线 | `MAX_RETRY`, `API_KEY` |
| 私有变量 | 单下划线前缀 | `_internal_data` |
